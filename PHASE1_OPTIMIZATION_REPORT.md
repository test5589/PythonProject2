# Phase 1 優化總結報告 (2026-05-15)

本報告記錄了 Web Charting 前端介面在第一階段（效能與穩定性優化）中的開發歷程、遇到的問題以及最終的解決方案。

## 1. 修改前後對比 (Before vs. After)

| 特性 | 最初狀態 (Unmodified) | 優化後狀態 (Optimized) |
| :--- | :--- | :--- |
| **更新機制** | 每次 1s 更新都執行 `setData` 全量重繪，導致閃爍。 | 採用「參數變動才重置，數據進入則增量更新」邏輯。 |
| **切換體驗** | 切換 Symbol/Interval 時舊資料殘留，等待期畫面混亂。 | 點擊切換立即清空圖表 (Instant Reset)，視覺乾淨。 |
| **穩定性** | 頻繁出現 `Cannot update oldest data` 或畫面全黑。 | 透過 `lastParamsRef` 嚴格控管 Series 生命週期，極度穩定。 |
| **Console 負擔** | F12 控制台每秒狂跳 Log，難以偵錯且影響效能。 | 實作 `Silent Mode`，監控模式下自動靜音，保持清爽。 |
| **數據連續性** | 1s 數據常因時差或後端限制出現空窗（全黑）。 | 後端改用動態時間窗 (Dynamic Window)，確保抓到最新蠟燭。 |
| **1m 模式更新** | 停止監控後 1m 圖表不再更新，除非手動重新整理。 | 加入 30s 背景自動刷新，確保非監控狀態下圖表依然在動。 |

---

## 2. 崩潰紀錄與修復歷程

### **A. 程式碼衝突導致的崩潰**
- **錯誤訊息**: `Identifier 'applyAutoWindow' has already been declared`
- **原因**: 在多次迭代修復中，重複貼上了同名的輔助函數。
- **解決**: 清理冗餘代碼，統一封裝 `applyAutoWindow` 邏輯。

### **B. 資料結構錯誤 (Critical)**
- **錯誤訊息**: `Cannot update oldest data`
- **原因**: 當從 1s 切換到 1m 時，試圖在同一個 Series 物件上餵入時間戳更早的資料，違反了 Lightweight Charts 的規則。
- **解決**: 在 `useEffect` 偵測到參數變化時，強制執行 `chart.removeSeries` 並清空 `Map`，確保每個 Series 都是全新的。

### **C. 監控啟動後的「全黑畫面」**
- **原因**: 
    1. 前端請求的 200 根蠟燭在 1s 模式下僅代表 3 分鐘，若後端有幾秒延遲，請求範圍就會剛好落在「數據尚未產生」的真空區。
    2. 後端使用硬編碼的 `datetime.now()` 減去秒數，若伺服器與資料庫時鐘不同步會導致抓不到資料。
- **解決**: 
    - 前端將 1s 抓取量提升至 500 根。
    - 後端 `charts.py` 改為先查詢 DB 的 `max_timestamp`，以該時間為基準回推，徹底消除「時間真空」。

## 3. 錯誤思路總結 (Incorrect Approaches)

在優化過程中，我們曾嘗試過以下「較不理想」的方案：

1. **試圖在不刪除 Series 的情況下相容數據**:
    - *錯誤想法*: 想透過過濾掉舊數據來避免 `Cannot update oldest data`。
    - *結果*: 導致切換後的圖表座標軸錯亂，且邏輯極度複雜。
    - *教訓*: 「切換即重置」是最乾淨且符合圖表套件設計邏輯的做法。

2. **過度依賴 `setData` 解決所有問題**:
    - *錯誤想法*: 認為只要每秒 `setData` 就能保持最新。
    - *結果*: 導致使用者無法在 1s 模式下自由縮放（因為每秒都會被強制重置視窗）。
    - *教訓*: 必須區分「初次載入」與「監控更新」的視窗控制權。

3. **嚴格的時間窗過濾**:
    - *錯誤想法*: 在後端 API 嚴格限制 `now() - N 秒`。
    - *結果*: 只要數據來源稍有延遲，圖表就直接斷頭（全黑）。
    - *教訓*: API 應該具備彈性，以「目前擁有的最新數據」為基準才是正確的。

## 4. 核心技術優化說明

1. **Ref-Based 狀態追蹤**: 使用 `lastParamsRef` 記錄上一次的 Symbol 與 Interval，這比 React State 更即時且不會觸發額外渲染，是精準控制圖表重置的關鍵。
2. **增量合併算法**: 在 `useCandles` 內實作了高效的 `Map` 合併邏輯，確保即使 API 回傳重複的數據，前端也能絲滑地銜接最新的一根蠟燭。
3. **自動視窗恢復機制**: 實作了「5秒無操作自動恢復」邏輯。使用者可以手動拉動查看歷史，但只要回到最新位置且靜止 5 秒，系統就會接管並繼續追蹤最新價格。

---

## 5. 結論

第一階段的優化成功將 Web Chart 從「可運作」提升到了「穩定專業」的程度。消除了所有已知的崩潰風險，並在視覺上達成了類似專業交易軟體的穩定感。
