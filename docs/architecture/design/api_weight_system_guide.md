# API 權重評估系統使用指南

## 📋 系統概述

根據 Blueprint.md 設計的 API 權重評估系統，用於防止因 API 請求過頻繁導致 IP 被鎖定。系統包含：

- **6個獨立時間框架**：1m, 3m, 5m, 10m, 30m, 1h
- **智能權重調整**：根據被鎖歷史動態調整請求筆數
- **雙循環機制**：第一循環處理連續被鎖，第二循環進行平均補償
- **完整日誌記錄**：所有權重變化都會記錄

---

## 🎛️ GUI 控制按鈕

### 📊 顯示權重狀態
**功能**：顯示所有 6 個時間框架的當前權重狀態

**輸出格式**：
```
📊 === API 權重評估狀態 ===
🟢 [1m] 權重=1.000 | 筆數=2000 | 狀態=normal | 總鎖定=0 | 解鎖=0
🟢 [3m] 權重=1.000 | 筆數=2000 | 狀態=normal | 總鎖定=0 | 解鎖=0
...
📊 權重狀態顯示完成
```

**狀態圖示**：
- 🟢 normal：正常狀態
- 🔒 locked：已被鎖定
- 🔓 unlocked：已解鎖
- 🧠 second_cycle：第二循環處理中
- ⛔ halted：權重過低，已中止操作

---

### 🔄 重置所有權重
**功能**：將所有時間框架重置為預設值

**預設值**：
- 權重：1.0
- 基礎筆數：2000
- 清除所有被鎖紀錄

**注意**：此操作會彈出確認對話框，確認後才會執行。

---

### 🧪 權重測試模式
**功能**：模擬 API 被鎖定情況，測試權重調整邏輯

**測試流程**：
1. 模擬 1m 時間框架的連續被鎖
2. 依序測試：7584 → 6067 → 5157 → 4641 → 4000 筆數
3. 自動觸發第一循環和第二循環
4. 顯示最終權重狀態

**注意**：此操作僅為測試，不會影響真實 API 請求。

## 🔄 權重調整邏輯

### 第一循環（被鎖處理）
當 API 被鎖時觸發：

1. **扣除比例序列**：20% → 15% → 10% → 20%（循環）
2. **權重計算**：
   - 被鎖時：權重 -= 扣除比例
   - 解鎖時：權重 += (扣除比例 - 0.01)
3. **筆數調整**：基礎筆數 × (1 - 扣除比例)
4. **連續判定**：連續 5 次被鎖觸發第二循環

**範例**：
```
初始：權重=1.0, 筆數=2000
第1次被鎖：扣除20% → 權重=0.8, 筆數=1600
解鎖後：權重=0.8+0.19=0.99
第2次被鎖：扣除15% → 權重=0.84, 筆數=1360
```

---

### 第二循環（平均補償）
當第一循環滿 5 次被鎖時觸發：

1. **計算平均扣除**：最近 5 次的平均扣除比例
2. **補償公式**：
   - X = 平均扣除 × 0.312
   - Y = (平均扣除筆數) × X
3. **權重恢復**：當前權重 + X
4. **筆數恢復**：當前筆數 + Y

**終止條件**：
- 權重 > 1：恢復正常，結束循環
- 權重 < 0.81：中止所有操作

## 🔧 程式碼整合

### 在 API 請求中使用
```python
from modules.api_weight_evaluator import get_api_weight_evaluator

# 取得建議請求筆數
evaluator = get_api_weight_evaluator()
count = evaluator.get_optimal_count("1m")

# 執行 API 請求
try:
    result = api_request(count=count)
except RateLimitError as e:
    # 標記被鎖
    evaluator.mark_lock("1m", count)
    # 等待解鎖後
    evaluator.mark_unlock("1m")
```

### 使用上下文管理器
```python
with evaluator.api_request_context("1m", requested_count=5000) as count:
    result = api_request(count=count)
# 自動處理被鎖情況
```

## ⚠️ 注意事項

1. **時間框架獨立性**：每個時間框架的權重設定獨立，不互相影響
2. **資料持久化**：所有權重變化會自動保存到 JSON 檔案
3. **執行緒安全**：系統使用鎖保護共享資源，支援多執行緒環境
4. **日誌記錄**：重要操作都會記錄到日誌檔案，便於問題追蹤

1. **定期監控**：使用「顯示權重狀態」按鈕定期檢查各時間框架狀態
