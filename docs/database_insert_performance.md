# 資料庫插入性能分析

## 🐌 問題描述
用戶報告資料插入速度慢。

## 🔍 性能問題分析

### 當前實現：單筆插入 ⚠️ 效率低

**位置**: `modules/utils/backfill/backfill_data.py:225`
```python
# 每筆資料都單獨呼叫 insert_data
success, meta = insert_data(
    category, symbol, interval_seconds, k, 
    data_source='real', ...
)
```

**每次插入的操作**：
1. 獲取資料庫連接（從連線池）
2. 開始隱式事務
3. 執行 SELECT 檢查是否存在
4. 執行 INSERT OR REPLACE
5. **提交事務** ⚠️
6. 返還連接到連線池

**性能影響**：
- 999筆資料 = 999次完整的資料庫操作週期
- 999次事務提交（磁碟I/O密集）
- 999次連接池操作
- SQLite的COMMIT非常慢（需要fsync到磁碟）

### 批量插入實現：已存在但未使用 ✅

**位置**: `modules/utils/database/data_manager.py:339`
```python
def batch_insert_data_optimized(...):
    conn.execute("BEGIN TRANSACTION")  # 只開始一次
    
    for kline in klines:
        # 插入每筆資料（不提交）
        self._insert_single_in_transaction(...)
    
    conn.commit()  # 只提交一次！
```

**優勢**：
1. 只獲取一次連接
2. 所有插入在同一個事務中
3. **只提交一次**（關鍵！）
4. 只返還一次連接

## 💡 為什麼COMMIT這麼慢？

SQLite的COMMIT操作：
1. 將數據寫入WAL文件
2. **調用fsync()強制寫入磁碟** ⚠️ 最慢
3. 更新資料庫檔案頭
4. 可能需要checkpoint

在機械硬碟上，每次COMMIT可能需要5-10ms
在SSD上，也需要1-5ms

**結論**：減少COMMIT次數是提升性能的關鍵！

## ⚠️ 注意事項

1. **進度報告**：批量插入時無法即時報告每筆進度
   - 解決：報告批次進度而非單筆進度

2. **跳過計數**：批量插入需要調整邏輯
   - 解決：使用批量插入的返回值

3. **錯誤處理**：批量失敗時需要回滾
   - 解決：已有事務回滾機制

4. **記憶體使用**：批量可能佔用更多記憶體
   - 影響：999筆資料約1-2MB，可接受
