# 修復總結 - 2025/11/11

## 🐛 修復的問題

### 1️⃣ **回補 1 小時後出現 1s 資料回補**
**問題原因**：
- 回補完成後會自動執行 `slow_background_seconds()` 函數
- 該函數會自動補充 1 秒級別資料

**解決方案**：
- 移除自動啟動 1s 回補的程式碼
- 如果需要 1s 資料，用戶應該使用「啟動 1秒監控」按鈕

**修改檔案**：
```python
# core/Gui.py - 第 336 行
# 移除自動啟動 1s 回補，避免用戶混淆
# threading.Thread(target=slow_background_seconds, args=(cat, sym, gui_safe_log), daemon=True).start()
```

**問題原因**：
- GUI 主視窗：800x600 太小

### 3️⃣ **連續按兩次回補未出現覆蓋提示**
**問題原因**：
- 第一次回補設定覆蓋偏好後，該設定會持續到 GUI 關閉
- 第二次回補時，系統直接使用上次的偏好設定，不再詢問

**解決方案**：
- 每次開始回補時重置 `overwrite_real_preference` 和 `overwrite_asked_count`
- 確保每次回補都從頭開始詢問

**修改檔案**：
```python
# core/Gui.py - backfill_data() 函數
# 每次開始回補時重置覆蓋偏好（確保每次都會詢問第一筆）
self.overwrite_real_preference = None
self.overwrite_asked_count = 0
```

### 4️⃣ **按暫停按鈕後當機**
**問題原因**：
- 原本的暫停實現在持有鎖的情況下等待，造成死鎖
- `threading.Event().wait(0.1)` 在鎖內執行，導致其他線程無法訪問狀態

**解決方案**：
- 改進鎖定機制：先更新狀態並釋放鎖，再檢查暫停狀態
- 暫停檢查移到鎖外執行，避免死鎖

**修改檔案**：
```python
# modules/utils/backfill_state.py - update_progress() 函數
def update_progress(self, index: int, message: str = ""):
    with self._lock:
        # 更新狀態
        self.state.current_index = index
        # ...
    
    # 在釋放鎖之後檢查暫停狀態
    while True:
        with self._lock:
            if self.state.is_stopped:
                raise InterruptedError("回補已被停止")
            if not self.state.is_paused:
                break
        time.sleep(0.1)  # 在鎖外休眠
```

## 🎯 使用建議

### **回補不同時間段的資料**
1. **短期回補（幾小時到幾天）**：
   - 使用 1分、5分、15分
   - 資料量適中，回補速度快

2. **中期回補（幾天到幾週）**：
   - 使用 30分、1小時
   - 平衡資料量和細節

3. **長期回補（幾週到幾個月）**：
   - 使用 4小時、8小時、12小時、24小時
   - 減少資料量，加快回補速度

### **1 秒資料的獲取**
- ⚠️ **不要使用回補功能**獲取 1 秒資料
- ✅ 使用「🟢 啟動 1秒監控」按鈕
- ✅ 1 秒資料只能即時獲取，不支援歷史回補

## ⚠️ 重要提醒

1. **每次開始回補都會重置覆蓋設定**
   - 即使上次選擇「覆蓋所有」，下次回補還是會詢問

2. **暫停記錄在記憶體中**
   - 關閉 GUI 後暫停位置會遺失
   - 重新開啟 GUI 需要重新回補

3. **完全停止不保存進度**
   - 點擊「⏹️ 完全停止回補」會立即終止
   - 已處理的資料會保存，未處理的放棄

4. **間隔選單只影響回補功能**
   - 「📥 抓取最新資料」固定使用 1 分鐘
   - 「🟢 啟動 1秒監控」固定使用 1 秒
